# What Good Looks Like

A strong answer in this lab should:

- identify the main package modules and what each one does
- mention the test file as a source of truth for expected behavior
- describe the main data shape used by the project
- point out likely extension points without touching code
- stay read-only and avoid inventing work that was not requested

Weak answers usually:

- skip the tests
- jump to refactoring ideas too early
- suggest edits before understanding the repo
